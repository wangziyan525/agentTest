# UI 1:1 Restore Flow

## 目标

把 `ui/` 导出包里的单个 screen 稳定落到 `ares/` 对应页面中，尽量做到 1:1 还原，而不是靠主观试错反复调。

## 适用任务

- 根据 `ui/links`、`ui/preview`、`ui/assets`、`ui/index.html` 还原页面
- 页面已经有初版，但和设计稿仍有差异，需要继续往 1:1 收
- 需要把“怎么对图、怎么落地、怎么验收”的方法沉淀成固定执行流

## 输入源优先级

1. `ui/preview/*.png`
   这是最终视觉主参照。
2. `ui/index.html`
   这是规格主参照，用来补齐坐标、尺寸、颜色、字号、圆角、切图名称。
3. `ui/assets/*`
   这是图标、按钮、状态图资源来源。
4. `ui/links/*.html`
   这是 screen 名称和跳转锚点来源。
5. `ares/` 同模块现有页面
   这是宿主技术栈和代码组织方式来源。

## 快速开始

### 1. 锁定目标 screen 和落地点

- 先从 `ui/links/` 或 `ui/preview/` 确认目标 screen 名。
- 再结合 `ares/<module>/README.md` 或现有页面命名确认落地点。
- 如果一个 screen 名里有：
  `备份`
  `成功`
  `失败`
  `拒绝`
  `待签收`
  这通常表示不同状态页，不要混成一个页面理解。

### 2. 先提取 artboard，再开始写代码

- 先从 `ui/index.html` 里提取目标 artboard。
- 至少拿到这些信息：
  - 画板宽高
  - 关键图层坐标
  - 文字内容和字号
  - 背景色和分隔线色
  - 切图资源名
- 不拿到画板规格，不允许直接开始“凭感觉”改布局。

推荐命令：

```bash
node skills/ui-1to1-restore-flow/scripts/list_artboards.js ui/index.html
node skills/ui-1to1-restore-flow/scripts/extract_artboard.js ui/index.html "<screen-slug>"
```

### 3. 先拆区域，再列对照项

先把页面拆成固定区域：

- 顶部栏
- 顶部主视觉区
- 信息行/卡片区
- 底部按钮区
- 灰底/白底承托区

每个区域至少要列出：

- 起止坐标
- 主背景色
- 主文案
- 资源图
- 高风险细节

最小对照清单至少包含：

- 画板宽高
- 背景分层
- 顶部栏位置和尺寸
- 关键图标尺寸和位置
- 信息行高度、分隔线、左右留白
- 按钮宽高、圆角、位置

### 4. 先对宿主骨架，再做像素收口

- 先确认目标模块技术栈、目录结构、公共样式和脚本引用。
- 保持 `ares` 现有 `html + css + js` 模式。
- 样式优先写页面级 class，避免污染同模块其他页面。

## 实现规则

- 如果 `ui/index.html` 已经给出明确 `x/y/width/height`，优先按这些值实现。
- 需要固定对齐的区域，不要优先用模糊的弹性分布去猜。
- 设计稿中明显属于同一编组的元素，例如：
  - 票号 + 状态标签
  - 标题 + 小图标
  - 金额 + 单位
  - 左文案 + 右箭头
  优先按画板坐标和固定间距还原，不要先套 `flex: 1`、`justify-content: space-between` 这类通用布局把它们推开。
- 图标、状态图、特殊箭头、按钮素材优先用 `ui/assets/` 中已有资源，不要手画近似替代物。
- 箭头、筛选、排序、折叠展开这类状态型 icon，不能只看资源名；必须回到 `ui/preview/*.png` 核对最终观感，再决定直接用、旋转使用还是用 CSS 重画。
- 页面背景按“承托层”理解：先看整页底色，再看白块、灰块、卡片和区块，避免漏掉灰底留白、按钮承托区、顶部过渡层。
- 文本不要只看字号，同时看字重、行高、颜色、文本框宽度和左右对齐边界。
- 实现顺序固定：先结构，再资源，再文本边界，最后收细节颜色和间距。

## 验收清单

- 结构是否对齐：顶部栏、主视觉区、信息区、按钮区是否分层正确。
- 资源是否正确：返回箭头、状态图、按钮图、右侧箭头是否用对，方向和当前态是否与 `preview` 一致。
- 坐标是否正确：文本左边距、右侧值收边、分隔线位置、按钮宽高与纵向位置是否一致。
- 承托层是否补齐：灰底/白底层次、区块间留白、顶部过渡、按钮承托区是否到位。
- 细节复核必须单独执行一轮，至少检查：
  - 箭头、小 icon、checkbox、radio、tag、禁用按钮
  - 分隔线、浅底承托层、卡片边界
  - 长文案换行、右对齐边界、数字金额收边
  - 选中 / 未选中 / 激活 / 禁用态
  - fixed 底栏、safe area、底部留白是否遮挡内容
- 只要有任一项无法和 `ui/preview/*.png` 对上，就不能表述为“已 1:1 还原”；无法确认时明确标记为“待确认”。

## 明确禁止事项

- 不要只看 preview 肉眼猜布局，不看 `ui/index.html`
- 不要改一轮就停，必须按清单至少做一轮自检
- 不要因为大结构已经对，就跳过细节复核
- 不要为了省事把整张预览图铺成背景图
- 不要让宿主页面的旧样式污染当前 screen，又不做页面级隔离

## 产出要求

- 页面落地代码
- 本次 screen 的对照清单
- 本次 screen 的细节复核结果
- 已确认一致的区域
- 仍待真机或浏览器截图确认的区域

## 快速模板

每次开始一个新 screen，先回答这 6 个问题：

1. 目标 screen slug 是什么？
2. 对应的 `ares` 页面文件是什么？
3. 画板宽高是多少？
4. 顶部栏、主视觉区、信息区、按钮区分别多高？
5. 需要用到哪些 `ui/assets` 资源？
6. 当前最可能漏掉的承托层是什么？
